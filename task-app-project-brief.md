# Проект: Self-hosted приложение за организиране на задачи

## Контекст и цел

Искам да изградя собствено уеб приложение за организиране на задачи (task manager), което:
- Ще хостна сам на стар компютър/лаптоп вкъщи (self-hosted, безплатно)
- Ще достъпвам от няколко устройства — Android и iOS телефони
- Ще позволява задаване на дата, час и бележки към всяка задача
- Ще се ползва от мен + партньор/семейство (2-4 души общо)

Аз имам основни познания по програмиране, но не съм писал backend преди. Моля структурирай проекта стъпка по стъпка, обяснявай ключовите решения накратко, и предпочитай прости, добре документирани решения пред сложни/оптимизирани такива.

## Технологичен стек (вече избран — не предлагай алтернативи, освен ако не попитам)

- **Backend**: Node.js + Express (плитка крива на учене, много документация)
- **База данни**: SQLite (един файл, лесен backup, достатъчен за 2-4 потребители)
- **Автентикация**: JWT токени, отделен акаунт за всеки от 2-4-те потребители
- **Frontend**: PWA (Progressive Web App) — React или Svelte (по избор на Claude Code, но с предпочитание към по-простия вариант с оглед моето ниво)
- **Контейнеризация**: Docker + Docker Compose (backend + база данни в един compose файл)
- **Достъп отвън**: Cloudflare Tunnel (`cloudflared`) — HTTPS без отваряне на портове на рутера, без нужда от статичен IP

## Защо този стек (контекст за решенията)

- PWA вместо native (React Native/Flutter/Swift) — защото native iOS изисква Apple Developer акаунт (99$/година) и Mac за компилиране; PWA работи еднакво на Android и iOS директно през браузър, безплатно.
- SQLite вместо PostgreSQL — при 2-4 потребители няма нужда от по-тежка база данни; файлът се архивира лесно.
- Cloudflare Tunnel вместо port forwarding — по-сигурно (не отваряш портове към домашната мрежа) и не изисква статичен IP от интернет доставчика.
- Express вместо по-сложни framework-и — най-достъпен за някой с основни познания по програмиране.

## Функционални изисквания

1. Създаване на задача с: заглавие, дата, час, бележки/описание, статус (незавършена/завършена)
2. Редакция и изтриване на задачи
3. Изглед по ден/седмица (календарен изглед е желателен, но не критичен за MVP)
4. Логин система — всеки от 2-4-те потребители има собствен акаунт
5. Възможност за лични задачи и (по избор, по-късна фаза) споделени задачи между потребителите
6. Работи офлайн (кешира последно заредените задачи) и се синхронизира при връзка
7. Push известия/напомняния за задачи — важно: тества се рано, тъй като поддръжката на iOS за PWA push известия е ограничена и по-нова функционалност

## Технически изисквания за PWA (задължителни за инсталиране на телефон)

- **`manifest.json`** — с име на приложението, икони (различни размери), theme_color, display: "standalone"
- **Service Worker** — за кеширане на assets и офлайн работа
- **HTTPS е задължителен** — както Android, така и iOS отказват инсталация на PWA без валиден HTTPS сертификат (Cloudflare Tunnel решава това автоматично)
- Инсталация на Android: чрез Chrome, банер "Add to Home screen" или меню → "Install app"
- Инсталация на iOS: само през Safari → Share бутон → "Add to Home Screen" (Chrome на iOS не поддържа тази функция)

## План на действие / фази на разработка

Моля следвай тези фази последователно, потвърждавай завършването на всяка преди да преминеш към следващата:

### Етап 1 — Подготовка на средата
- Инсталация на Linux (ако компютърът все още няма) на стария лаптоп
- Инсталация на Docker и Docker Compose
- Инсталация на Node.js (за локална разработка)

### Етап 2 — Backend и база данни
- Проектиране на SQLite схема: users, tasks (title, date, time, notes, status, user_id, shared boolean)
- Express REST API: CRUD endpoints за задачи, endpoint за логин/регистрация с JWT
- Основна валидация на входните данни

### Етап 3 — Frontend (PWA)
- React или Svelte приложение с изглед по ден/седмица
- Форма за създаване/редакция на задача (дата, час, бележки)
- Логин екран
- `manifest.json` и Service Worker за PWA функционалност

### Етап 4 — Deployment
- Docker Compose файл, който пуска backend + база данни заедно
- Настройка на Cloudflare Tunnel за HTTPS достъп отвън
- Тест на инсталация като PWA от Android и iOS устройство

### Етап 5 — Потребители и споделяне
- Отделни акаунти за всеки от 2-4-те потребители
- (По избор) споделени задачи, видими за всички в семейството

### Етап 6 — Полиране
- Push известия / напомняния (приоритет: тества се рано заради ограниченията на iOS)
- Автоматични backup-и на SQLite файла
- Дребни UX подобрения

## Как да подходиш

- Стартирай директно от Етап 1, освен ако не поискам друго
- Обяснявай кратко всяко ключово решение, без да предполагаш, че знам backend концепции отнапред
- Предпочитай ясен, добре коментиран код пред "elegant"/сбит код
- Не предлагай алтернативен технологичен стек — той вече е избран по-горе; ако имаш резерви, спомени ги накратко, но продължи с избрания стек, освен ако изрично не поискам промяна

## Анализ и корекции (добавено от Claude, 2026-07-20)

Прегледах брифа — стекът е разумен за 2-4 потребители. Няколко пропуска, които допълвам, без да сменям избрания стек:

1. **Регистрация**: няма да имплементирам публичен `/register` endpoint. Приложението ще е достъпно от интернет през Cloudflare Tunnel, а брой потребители е фиксиран (2-4) — открита регистрация е ненужен риск. Вместо това: CLI seed скрипт, с който ти създаваш акаунтите ръчно.
2. **Пароли**: ще се хешират с bcrypt (bcryptjs — чист JS, без native compile проблеми на Windows), никога plain text.
3. **JWT съхранение**: токенът ще живее ~30 дни (просто решение вместо access+refresh token двойка — приемлив компромис за доверено семейно приложение, но означава, че открадната сесия важи дълго; ще пазим `JWT_SECRET` само в `.env`, не в код).
4. **CORS/архитектура**: Express ще сервира и статичния build на PWA-то в production — един процес, един Docker контейнер, без CORS проблеми. CORS се включва само в dev режим, докато frontend dev сървърът работи на отделен порт.
5. **SQLite персистентност**: файлът трябва да е на Docker volume, иначе се губи при пресъздаване на контейнера — ще адресирам в Етап 4.
6. **Push известия**: изискват VAPID ключове + `web-push` библиотека на бекенда — ще е нужна допълнителна настройка в Етап 6, отделно от Service Worker-а от Етап 3.
7. **Frontend framework**: избирам **Svelte** (не React) — по-малко boilerplate, реактивност без JSX/hooks, по-близко до обикновен HTML/CSS/JS. По-подходящо за някой без backend/frontend опит.
8. **Етап 1 бележка**: инсталацията на Linux/Docker на стария лаптоп е физическа задача на твоята машина — не мога да я извърша от тук. Node.js и Docker вече присъстват на текущата машина, така че стартирам directно с Етап 2 (код, който после ще се деплойне на лаптопа чрез Docker).
